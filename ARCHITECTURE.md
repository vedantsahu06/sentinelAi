# Sentinel AI - Architecture & Development Plan

## Overview
Sentinel AI is an AI-powered Incident Investigation and Organizational Memory Platform. It acts as a persistent incident memory layer for engineering teams, where it investigates incidents, remembers previous ones, learns from resolutions, and improves over time.

## 1. Folder Structure
The codebase follows clean architecture principles, separating the agent logic, tools, integrations, and soon the frontend/backend services:

```
├── agent_system/             # Core LangGraph AI logic
│   ├── graphs/               # LangGraph state graphs
│   ├── nodes/                # Individual LangGraph nodes
│   ├── state/                # Strongly-typed state definitions
│   ├── prompts/              # System prompts for LLMs
│   └── routers/              # Routing logic if needed
├── tools/                    # Tools utilized by the AI Agent
│   ├── log_parser.py
│   ├── incident_classifier.py
│   ├── memory_search.py
│   ├── root_cause_investigation.py
│   ├── recommendation.py
│   ├── confidence_scoring.py
│   └── memory_writer.py
├── integrations/             # External service wrappers
│   ├── llm/
│   └── postgres/
├── prisma/                   # Database schema
│   └── schema.prisma
├── main.py                   # Entry point for running the graph
└── ARCHITECTURE.md           # This document
```

## 2. LangGraph Workflow (Graph)

The investigation process is modeled as a deterministic LangGraph state machine.

**Flow:**
`START` -> `log_ingestion` -> `log_summarization` -> `incident_classification` -> `memory_retrieval` -> `historical_incident_analysis` -> `root_cause_investigation` -> `recommendation_generation` -> `confidence_scoring` -> `human_feedback` -> `memory_persistence` -> `END`

## 3. State Definitions
The state is strongly typed using Python's `TypedDict` in `agent_system/state/sentinel_state.py`.
It contains:
- `incident_id`: str
- `raw_logs`: str
- `parsed_logs`: List[dict]
- `incident_type`: str
- `severity`: str
- `service_name`: str
- `similar_incidents`: List[dict]
- `retrieved_memories`: List[dict]
- `root_cause`: str
- `recommendations`: List[dict]
- `confidence_score`: float
- `human_feedback`: dict
- `final_resolution`: dict
- `memory_written`: bool

## 4. Tools Implemented
1. **Log Parser Tool**: Parses raw text logs into structured dictionaries.
2. **Incident Classifier Tool**: Categorizes incidents (database, auth, network, memory, etc.) based on keywords.
3. **Memory Search Tool**: Searches Hindsight memory for semantically similar past incidents.
4. **Root Cause Investigation Tool**: Analyzes logs and historical context to deduce RCA.
5. **Recommendation Tool**: Generates fixes, investigations, and rollback steps based on RCA.
6. **Confidence Scoring Tool**: Evaluates the match against history to provide a 0-100 score and reasoning.
7. **Memory Writer Tool**: Stores structured incident knowledge (RCA, fix, outcome) back to the database.

## 5. Database Schema
Defined using Prisma in `prisma/schema.prisma`.
Tables included:
- **Users**: Engineers interacting with the system.
- **Incidents**: The core incident record.
- **IncidentMemories**: Structured knowledge of past incidents.
- **Recommendations**: Suggested fixes generated by the agent.
- **Resolutions**: Confirmed fixes and outcomes.
- **Feedback**: Human feedback on resolutions.
- **AgentRuns**: Metadata on LangGraph executions.
- **IncidentEmbeddings**: pgvector definitions for semantic search.

## 6. APIs (Proposed for Backend)
* `POST /api/incidents`: Ingest new incident & trigger LangGraph workflow.
* `GET /api/incidents/{id}`: Fetch incident details and current agent run status.
* `POST /api/incidents/{id}/feedback`: Submit human feedback (triggering the `human_feedback` node via interrupt).
* `GET /api/memory/search`: Search historical incidents.
* `GET /api/analytics/learning`: Fetch data for the Organizational Memory Dashboard to visualize Sentinel's learning curve.

## 7. Development Roadmap

**Phase 1: Agent Core (Completed)**
* Design LangGraph state and nodes.
* Implement core tool stubs (Mocked Semantic Search, LLM parsing).
* Connect the execution graph.
* Define Prisma Schema.

**Phase 2: Real Integrations**
* Connect `memory_search` to PostgreSQL `pgvector`.
* Connect LLM nodes to Groq/Langchain for genuine log analysis instead of heuristics.
* Implement Next.js + FastAPI boilerplate.

**Phase 3: Frontend & Hackathon "Wow" Features**
* Build Dashboard (SRE style, Dark Mode).
* Implement Incident Timeline and Resolution Replay UI.
* Implement Organizational Memory Dashboard showcasing continuous learning.
* Build the Demo Flow to show the exact 3-incident scenario described in the requirements.
